# SOLO/RQS バリデーター

teachAI 評価エンジンの整合性を守るためのコード/データ検査スキル。

## 検査軸

### A. SOLO Taxonomy(1-5)の整合性
- 1: 前構造的(無関連)
- 2: 単一構造(1要素のみ)
- 3: 多重構造(複数要素並列)
- 4: 関係的(統合)
- 5: 拡張抽象(他領域への適用)

**チェック項目:**
- スコア計算で SOLO レベルの順序が単調に効いているか
- レベル境界の定義が明示されているか(機械的に判定可能か)
- レベルアップは「項目追加」だけでなく「関係性の出現」を要件にしているか

### B. RQS(Recursive Question Scoring)の整合性
- 質問の連鎖が深くなるほど高評価
- 同レベルの質問を繰り返すだけでは加点しない

**チェック項目:**
- 質問深度の計測ロジックが「質問の親子関係」を見ているか
- 単なる「質問数」を加点していないか
- ユーザーが自分で質問を発する場合の加点ルールが定義されているか

### C. Elo レーティングの整合性
- 各トピックに Elo 値を持たせ、検算成績で更新
- K-factor が固定値で安定しているか

**チェック項目:**
- Elo 更新式が対称(勝者+a, 敗者-a)になっているか
- 急激な変動を抑える上下限が設定されているか
- 「教える側 vs AI生徒」のどちらが勝者/敗者かの定義が明示されているか

### D. 知識グラフ更新の整合性
- ノード = 概念、エッジ = 関係
- 検算セッションごとにノード追加・エッジ強化

**チェック項目:**
- 既存ノードの重複検出(同義語マージ)があるか
- エッジの強度更新が単調か(検算成功で上昇、忘却で減衰)
- グラフが切断されないよう、孤立ノードを検出しているか

### E. 説明可能性の保持
- スコア表示には必ず教材引用 + 内訳 + 改善ヒントが添えられる

**チェック項目:**
- スコア生成関数の戻り値に `rationale` フィールドがあるか
- `rationale` が空配列のままユーザーに表示されるパスがないか
- フロントエンドが `rationale` を画面に出しているか

## 検査プロセス

1. 対象のファイル/PR/関数を読む
2. 上記 A-E の観点で問題があるかリストアップ
3. 問題ごとに「重大度: 高/中/低」と「修正案」を出す
4. 全体としての合否を出す

## 出力フォーマット

```
# SOLO/RQS バリデーション結果

対象: <ファイル名 / PR / 関数名>

## A. SOLO 整合性
- [問題] ... / [修正案] ... / [重大度] 高/中/低
- 問題なし

## B. RQS 整合性
...

## C. Elo 整合性
...

## D. 知識グラフ整合性
...

## E. 説明可能性
...

総合判定: 合格 / 要修正 / 却下
ブロッカー: <要修正以上の項目を列挙>
```

## 呼ばれ方

- 評価エンジン関連の PR レビュー時
- assessment-validator サブエージェントの中で
- 新しいスコアリングロジックを追加するとき
- データ移行時(過去スコアの再計算が走るとき)

## 重要

このスキルは「評価の権威」を担保する。説明可能性が落ちている変更は、機能的に正しくても却下する。
