# 機能仕様書テンプレ

teachAI の新機能は、コードを書く前に必ずこの仕様書を 1 ページ作る。

## テンプレ

```markdown
# Feature Spec: <機能名>

作成日: YYYY-MM-DD
作成者: はた
ステータス: ドラフト / 審査中 / 承認 / 却下 / 実装中 / 完了 / 撤退

## 1. WHY
なぜこの機能か。北極星「人間の思考力を取り戻す」とどう接続するか。
1-3 行で。

## 2. 北極星との接続証明
- 検算文化を強化する点: ...
- ユーザーが主権者でいられる理由: ...
- 説明可能性が保たれる方法: ...
- ユーザーデータ主権が保たれる方法: ...
- 10 年後も価値が残る理由: ...

(これは polaris-5q の事前回答)

## 3. アンチバリュー回避
- §3.4 のうち、抵触リスクのある項目を 1-9 から書き出す
- 各リスクに対する回避策を明示
- 「該当なし」とだけ書かない

## 4. UX 概要
- 主要画面 / 主要フロー
- ユーザーが見るもの(画像でもテキストでも)
- 説明可能性: スコア・推薦・通知に根拠を添える方法

## 5. 技術設計
- Supabase スキーマ変更の有無 / 内容
- API エンドポイント追加 / 変更
- 評価エンジンへの影響(SOLO/RQS/Elo/グラフ/忘却曲線)
- フロントエンド変更箇所
- 環境変数追加の有無

## 6. 影響範囲
- 既存ユーザーへの影響: 自動マイグレーション / 手動移行 / 互換性破壊
- データ移行計画(必要なら)
- パフォーマンス影響(クエリ数、レスポンス時間)

## 7. テスト計画
- 単体テスト: ...
- 統合テスト: ...
- 評価エンジン整合性: assessment-validator で確認
- キャラクター整合: character-consistency で確認

## 8. 撤退条件
- リリース後 N 週間以内に「ユーザーから X という反応が得られなければ撤退」
- 具体的な数値・観察指標

## 9. 工数見積
- 設計: X 日
- 実装: Y 日
- テスト・QA: Z 日
- 合計: N 日

## 10. リスク・未解決事項
- ...

## 11. 審査記録
- anti-value-guardian: 通過 / 却下 (YYYY-MM-DD)
- cold-consultant: 採用 / 改修要 / 却下 (YYYY-MM-DD)
- 最終判定: ...
```

## 作成プロセス

1. ユーザー(はた)から機能アイデアを受け取る
2. テンプレに沿ってドラフト
3. 不明箇所は「未定」と明記(空欄禁止)
4. anti-value-guardian に渡す
5. (通過なら)cold-consultant に渡す
6. 最終承認なら GitHub Issue として登録

## NG パターン

- 「WHY」欄に「面白そうだから」と書く
- 「アンチバリュー回避」を「該当なし」だけで済ます
- 「撤退条件」を書かない
- 工数見積を 1 日と書いて実際は 2 週間
- テスト計画を「動作確認」だけで終わらせる

## OK パターン

- WHY: "ユーザーが自分の説明を 1 週間後に再読したくなる仕組みを提供することで、検算習慣の定着を支援する"
- 撤退条件: "リリース後 4 週間で、再検算セッションの開始率が 5% を超えなければ撤退"
- 説明可能性: "スコア表示画面の右側に、参照した教材引用 + SOLO レベルの内訳 + 改善ヒントを常時表示"

## 重要

この仕様書は実装の指示書ではなく、「やる/やらないを判定するための資料」。
コードを書き始める前に審査を通すことで、半完成コードの廃棄を避ける。
