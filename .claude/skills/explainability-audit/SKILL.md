# 説明可能性監査

「ブラックボックスの権威に頼らない」を担保するためのスキル。

## 説明可能性の3要素

ユーザーが目にする判断には、以下の3つが必ず含まれていなければならない。

1. **根拠源** — 何を参照したか(教材の引用箇所、過去の発言、知識グラフのノード)
2. **計算の中身** — どう導出したか(SOLOレベル、RQS加点、Elo差分の内訳)
3. **次の行動の指針** — どうすれば改善できるか(検算ヒント、関連概念)

3つ揃っていないと、ユーザーは「なぜそう言われたか」が分からず、検算文化が崩れる。

## 検査対象

- スコア表示画面
- フィードバック文(キャラクターの発話)
- 知識グラフの強度変化
- 推薦コンテンツ
- メール / プッシュ通知
- API レスポンス(`/api/score`, `/api/feedback` 等)
- レポート / エクスポート

## 検査プロセス

1. 対象のコード / レスポンス / 画面を確認
2. ユーザーが見る判断項目を列挙
3. 各判断項目について 3 要素が揃っているか確認
4. 欠落箇所と修正案を提示

## 出力フォーマット

```
# 説明可能性監査

対象: <画面名 / API / 関数>

## 検出した判断項目
1. <判断A: 例 "SOLOスコア 3.5">
   - 根拠源: 有 / 無
   - 計算の中身: 有 / 無
   - 改善指針: 有 / 無
   - 判定: 合格 / 違反

2. <判断B>: ...

## 違反箇所
- <ファイル名:行> ...

## 修正案
- <具体的なコード変更案 / コピー追加案>

総合判定: 合格 / 要修正
```

## 違反例

- スコア「7.2」だけが表示され、内訳がない → 違反
- 「次におすすめの教材」が並ぶが、なぜその順序か説明なし → 違反 + §3.4#6 抵触
- フィードバック「もう少しです」だけで、教材のどこと食い違ったか示さない → 違反
- API が `{ "score": 7.2 }` のみで `rationale` を返さない → 違反

## 合格例

- スコア表示の隣に「根拠: 教材§2.3 行5の "X" に対しあなたは Y と説明。差分は ...」
- フィードバックに「あなたはAをBと結びつけたが、教材ではCを介してつながっている」
- API レスポンスに `rationale: { source, breakdown, hint }` がある

## 呼ばれ方

- assessment-validator サブエージェントの中で
- フロントエンド表示変更時の PR レビュー
- 新しい通知パスを追加するとき
- メール / 通知の文面レビュー

## 重要

説明可能性は「親切心」ではなく「誠実さ」の問題。根拠を出さない時点で、teachAI が答え屋になる第一歩となる。
