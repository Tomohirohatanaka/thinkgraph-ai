# Polaris 5Q — 機能判定 5 問

§8.2 で定義された機能採否の判断 5 問。anti-value-audit を通過した提案にこのフィルタをかける。

## 5 つの問い

機能提案について、以下を順番に答える。各設問は **Yes / No / 判断保留** の三択。

### Q1. 検算 (self-verification) の場を強化するか?
- ユーザー自身が「自分が分かっていなかった」と気づく時間を増やすか?
- 単に情報を増やしたり効率化したりするだけなら No。

### Q2. ユーザーが意思決定の主権者として振る舞えるか?
- ユーザーの判断・解釈・統合を AI が代行していないか?
- AI が「おすすめ」「正解」「次にやるべきこと」を提示するなら No。

### Q3. 評価結果に根拠を提示できるか?
- スコア / 提案 / フィードバックに、教材引用・ロジック・トレースを添えられるか?
- ブラックボックスで「これが良い」とだけ言うなら No。

### Q4. ユーザーデータの主権をユーザーに保てるか?
- ユーザーがいつでもエクスポート・削除できるか?
- ベンダーロックインを誘発する設計なら No。

### Q5. 10 年後にも価値が残るか?
- 一過性のトレンド・短期的なエンゲージメント稼ぎではないか?
- 「いま流行ってるから」が主理由なら No。

## 判定ルール

- **Yes 5 つ** → 即実装候補
- **Yes 4 つ + No か保留 1 つ** → 改修案を作成して再審査
- **Yes 3 つ以下** → 却下

## 出力フォーマット

```
# Polaris 5Q 判定

提案: <提案の要約 1 行>
前段: anti-value-audit = 通過

Q1 検算強化: Yes / No / 保留
  根拠: ...
Q2 主権維持: Yes / No / 保留
  根拠: ...
Q3 説明可能性: Yes / No / 保留
  根拠: ...
Q4 データ主権: Yes / No / 保留
  根拠: ...
Q5 10年後の価値: Yes / No / 保留
  根拠: ...

Yes 数: X/5
結論: 採用 / 改修要 / 却下
推奨アクション: ...
```

## 判定の厳しさ

- 「Yes 寄り」は **保留扱い** にする。0.5 を Yes に切り上げない。
- 「将来的には」「ユーザーが慣れれば」は判断材料にしない。今のデザインで判定する。
- Q1 と Q2 のどちらかが No なら、ほぼ確実に北極星から外れる。残りの 3 問が Yes でも採用しない。

## このスキルを呼ぶタイミング

- anti-value-audit で通過した機能提案について、次のステップとして必ず呼ぶ
- 既存機能の大規模リファクタを始める前
- 競合の動向に対応するかどうか迷ったとき
- 個人の思いつきを実装する誘惑にかられたとき
